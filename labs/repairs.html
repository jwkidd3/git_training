git clone https://github.com/rm-training/lab-revert-me

Undo an edit

I released something into master weeks ago, modifying chapter1.md so that it had a silly line, “There was definitely some rain…”. I’d like to undo this change.

Help me out by finding these commit(s) in the log and reverting them.

🔎 In the real world, with a simple one-liner like this I would probably just create a branch and manually undo the change - commit - push - open a pull/merge request and be done with it. However, lets assume that this change is actually quite large and hard for me to do by hand.

To undo this change, we’re going to revert it (rather than reset, which modifies history). In order to revert the change we first have to find the commit(s).

Here are some examples of searching the log, to get you started:

# all commits that have changes to a file
git log --oneline -- chapter1.md

# all commits, including patch info
git log --oneline -p -- chapter1.md

# all commits that include the string "rain" in their patch
git log -G "rain" -- chapter1.md

# combine it all to help find those 2 commits:
git log --oneline -G "rain" -p -- chapter1.md

Found the two commit ids? Revert them, starting with the most recent commit.

$> git log --oneline -G "rain" -- chapter1.md

f8829e0 Rain?
096217b Raining in chapter 1

$> git revert f8829e0
$> git revert 096217b

⚠️ You might get a conflict if you revert in the wrong order. Just revert --abort and try again.

Revert my merged finale branch

Unfortunately, I also merged in my finale branch way too early – I want to undo that.

🔎 Because it is a merge into master I consider it “released”. My publisher has probably already cloned this repo and started looking at it, proposing edits in their own branch. Therefore this branch is in use by other team members (shared) and I should not use reset to repair master.

Look at the graph to get your bearings.

git log --oneline --graph --all

Create a new branch off master to contain this reverted changeset, call it revert-the-finale

git switch master
git branch revert-the-finale

Determine which commit(s) id you are going to revert.

git log --merges --grep "finale"

Remember, we are reverting a merge, which requires we pick a mainline to “remain on”.

# <c1> is to be replaced with the merge commit id you find up above...
git revert <c1> -m 1

To finish, view the log and the diff of the revert you just made. finale.md should no longer exist on your branch.

# changes in my branch
git diff master...revert-the-finale

# new commits on my branch
git log master..

When you’re satisfied with the change, merge your revert-the-finale branch into master, making sure to not allow a “fast forward”. Do this locally (not via a remote).

git switch master
git merge revert-the-finale --no-ff

Cherry Picking

Remember that work we undid in chapter1.md, where I had been writing about the “rain”? I just decided I don’t want to throw it out entirely - I’d like to copy that work out into a new branch so I can keep working on it in isolation.

Use cherry-pick to “copy” just those commits into a new branch.

To do this, make sure you have those two commit ids handy.

f8829e0 Rain?
096217b Raining in chapter 1

Create a new branch off main or master.

git switch main
git switch -c rain-changes

Within that branch, cherry-pick the two commits over – if you’re feeling bold, experiment with --no-commit here to “squash” the two changesets into one. Unlike revert, you’ll copy these in the order the commits were originally made.

git cherry-pick 096217b
git cherry-pick f8829e0

🔎 Squashing would look like:

git cherry-pick 096217b --no-commit
git cherry-pick f8829e0 --no-commit
git add .
git commit

View the log to see what changed:

git log --oneline --graph --all

🔎 Cherry picking only copied the changesets from those commits, it did not move or remove the original commits from the graph.

Reset to Squash

reset is a very powerful command, it controls the “three trees”, which just means that we can use it to do a combination of:

  1. Moving branch pointers
  2. Cleaning up our staging area
  3. Cleaning up our working directory

Use reset to “squash down” the original finale branch.

🔎 You do this by finding the new commit id you want the branch to point to. Also: consider whether you’ll use --hard or not.

I suggest you create a copy of the branch, rather than modifying finale (so you can try out the upcoming bonus, which expects finale not to have been squashed).

git switch finale
git log --oneline --all

git switch -c squashed-finale

# find the commit id, then...
git reset 7d41853 --soft

Finally, check the status - you should see all the changes still sitting in your staging area. This is handy! You can now simply commit to effectively squash these into a new, single commit object.

git status
git commit -m 'Finale (Squashed)'

🔎 git merge-base can be used to determine the latest shared ancestor of two branches. In this way, we could have run git merge-base master finale to determine which commit finale branched off from. However, since master has since moved forward - and finale was actually merged (then reverted) the merge-base would actually be the tip of the original finale branch.
