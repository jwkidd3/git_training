Cleaning up

Overview:

Being able to clean up changes in our staging area and working directory is typical of most daily workflows.

In the same my-profile project, practice making and undoing changes ‚Äì undoing commits, un-staging, etc.

Setup

Start from the previous my-profile repository you already created, otherwise you can create a new repository with a single file (and single commit).

If you need a new repository quickly:

git init my-profile
cd my-profile
touch profile.md
echo "#My Name" >> profile.md
git add .
git commit -m "My First Commit"

1. Make some edits

Make the following changes so that we can practice using restore:

  1. Create an empty file named junk
  2. Make an edit to your profile.md file
  3. Create an empty file named LICENSE
touch junk
touch LICENSE
# also: edit your profile.md file in your fav editor
# echo "Junk Edit" >> profile.md

Check your status, you should have three changes listed in your working directory.

$> git status

  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   profile.md

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	LICENSE
	junk

Stage and commit all the changes:

git add .
git commit -m 'Bad commit!'

2. Undoing the commit

View your log, then undo your last commit:

git status
git log --oneline

# this is the "undo last commit" command:
git reset HEAD^

Now view your log and check your status to see what happened:

git log --oneline
git status

You‚Äôll see that the last commit you made is no longer in the log and the changes you had committed are now back in your working directory (even staging was cleaned up). This is nice - git didn‚Äôt throw out your work!

Your graph went from this:

A---B---C---D
            ^
            main [HEAD]

To this:

A---B---C
        ^
        main [HEAD]

üîé If you had passed --hard as an option to reset HEAD^, git would have thrown out your work, cleaning up both the staging and working directory areas to match the previous commit.

--hard tells reset to clean all three areas (working directory, staging area, your branch) to match the version you are resetting to.

3. Un-staging

First, stage everything, you‚Äôll see the following changes staged:

$> git add .
$> git status

On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	new file:   LICENSE
	new file:   junk
	modified:   profile.md

We only really want to commit the new LICENSE file.

Un-stage the changes to profile.md and junk:

$> git restore --staged profile.md junk
$> git status

On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	new file:   LICENSE

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   profile.md

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	junk

The changes to profile.md and junk were taken out of staging ‚Äì junk went back to being untracked which is its original state, and profile.md changes just remained in the working directory.

Now commit only the LICENSE file and check the status:

$> git commit
$> git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   profile.md

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	junk

no changes added to commit (use "git add" and/or "git commit -a")

You should still see the two pending changes, profile.md and the untracked junk file.

Delete the untracked junk file.

rm junk

Now, throw out the change to your profile.md file using restore:

git restore profile.md

Where did that work go? This is one of the few times you might be at risk of losing ‚Äúwork in progress‚Äù. Luckily, if you‚Äôve ever staged that work or committed it, it‚Äôs very easy to recover.

Before moving on, check your status (you should be all clean) and the log:

git status
git log --oneline

4. Amending a Commit

Let‚Äôs add some content to the exciting LICENSE file.

You just created the file in your last commit, so instead of creating a new commit, use amend to apply additional changes into the prior commit.

üîé --amend can be used to both edit the last commit message and apply additional changes to the previous commit

Edit the LICENSE file ‚Äì add a title to the file contents.

Then stage those changes:

git add LICENSE

Now, instead of creating a new commit, amend the last commit:

git commit --amend

Your graph goes from this:

A---B---C---D
            ^
            main [HEAD]

To this:

A---B---C---F
            ^
            main [HEAD]

Check your status and your log ‚Äì your latest commit id should be new!

git status
git log --oneline

üîé Amending is a ‚Äúhistory altering‚Äù command, which is something we can do safely on commits that we have not yet shared with our team. However, once you push a commit (or branch) up, this implies the commit(s) are shared and you should no longer ‚Äúamend‚Äù those specific commits.

Bonus 1: Hard Resets

Git will generally avoid touching untracked files until you add them to the project.

This can sometimes be confusing when you use reset, and it can also be tedious when you‚Äôre actually trying to clean up a lot of untracked files you no longer want around.

Start by creating 3 (or more) files in your project. Then stage & commit them.

# create the files
touch 1 2 3

git add .
git commit -m 'All my junk'

Now, use reset HEAD^ to undo the commit you just made - then check your status to see what remains:

$> git reset HEAD^
$> git status

On branch main

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	1
	2
	3

Those files are back to being untracked. That is to be expected, because that was their original state at this point in history‚Ä¶ they are not present in your staging area.

Let‚Äôs try that again:

git add .
git commit -m 'All my junk (take 2)'

This time, use reset --hard to undo your last commit, then check the status:

$> git reset HEAD^ --hard

$> git status
On branch main

nothing to commit, working tree clean

--hard is another mode of reset, and it tells reset to clean everything up to match where you are resetting to‚Ä¶ so now your working directory, staging area and branch all match the previous version.

Extra Bonus: How can you get those files (1, 2, 3) back? Hint: reset can help.

The solution... 
# find the commit where you created those three files
# then reset back to it:
git reset ? --hard


Bonus 2: Squash Commits

Squashing is the act of melding multiple commits into fewer (or just one) commits.

üîé You might ‚Äúsquash‚Äù in order to clean up your work before sharing with the team so that it looks like you only did one or two commits and not 20-30.

It‚Äôs a nice way to create a focused set of commits, rather than a long chain of work that might contain a lot of mistakes, trial-and-error, typos, test breakages, etc.

Given what you know, you should be able to pull this off.

Create two changes (to profile.md) in two sequential commits, then attempt to ‚Äúsquash‚Äù these into just one commit.

Your log should go from:

$> git log --oneline


c12f218 (HEAD -> main) New Edit 1
90e230c New Edit 2
485290e License File
...

To:

$> git log --oneline

f47f294 (HEAD -> main) Combined Edit 1 & 2
485290e License File

...

Need a hint? You‚Äôll use reset to pull this off.

The solution... 
# you'll reset back to the starting commit
# making sure not to do a --hard reset
git reset 485290e

# then stage & commit forwards
git add .
git commit -m 'Squashed things...'
