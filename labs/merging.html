2. Comparing

What file changes will be introduced when we merge add-chapter-1? diff can help us see that.

Compare the state of the files in add-chapter-1 to main.

git diff main...add-chapter-1

üîé The diff .. operator would have shown the collective difference, while ... is just the difference in the second argument, add-chapter-1.

Then use log to see the new commits you‚Äôll be introducing when you merge:

3. Merging to ‚ÄúRelease‚Äù

Let‚Äôs consider the work in add-chapter-1 complete - therefore we‚Äôre going to ‚Äúrelease‚Äù it by merging it into our mainline branch, main.

üîé I‚Äôm using the word ‚Äúrelease‚Äù loosely here. All we‚Äôre doing is combining our work in add-chapter-1 with another branch.

For many teams, ‚Äúmerging into the mainline/trunk‚Äù will be performed on a platform like GitHub, GitLab, BitBucket or Gerrit, since we can require code reviews and trigger an automated pipeline to be run before changes are merged.

Make sure you‚Äôre on the main branch, then merge add-chapter-1 into main.

‚ö†Ô∏è Remember, merge affects the branch you are on.

git switch main
git merge add-chapter-1

What type of merge did it perform? Check your graph to see the outcome.

git log --oneline --graph

You should have seen a fast forward merge. If not, this means you did some new work on main as well. It‚Äôs not terrible, but generally bad practice when collaborating with a team.

5. Release ‚ÄúChapter 2‚Äù

Let‚Äôs assume our work on chapter2.md is also considered done. Merge it into main to release it.

git merge add-chapter-2

What type of merge did it perform?

View the graph: git log --oneline --graph --all.

You should have seen a recursive merge happen, which requires a commit message (from you) and creates a new ‚Äúmerge commit‚Äù.

Inspect the new merge commit - does it contain any file changes? There won‚Äôt be changes, unless there was a conflict that had to be resolved with that merge. A merge commit just ties two (or more) histories together.

git show HEAD

Can you determine which commit ids are the ‚Äúfirst parent‚Äù and ‚Äúsecond parent‚Äù of the merge commit you just created?

# inspect the second parent

# look at how the commit stores the list of "parents"

6. Branch Cleanup

View which branches we‚Äôve merged and which still have work to merge in. We can filter the branch list like so:

git switch main
git branch --merged
git branch --no-merged

üîé Filtering the branch list like this is actually checking for branches that are merged into your current branch. So, if you aren‚Äôt ‚Äúon‚Äù main then you‚Äôll possibly see different results.

Both add-chapter-1 and add-chapter-2 are merged.

Delete the merged branches:

git branch -d add-chapter-1 add-chapter-2

What about the-intro? What happens if you attempt to delete it with -d?

git branch -d the-intro

With the lowercase -d flag, git will prevent you from accidentally deleting a branch that is not yet merged into your current branch.

üîé Do not actually delete the-intro. Just FYI: You can force delete a branch with -D ‚Äì but don‚Äôt try this just yet! We still want to work on the-intro

7. Get the-intro up to date

The the-intro branch is now a little behind main ‚Äì there are changes in main that are not in the-intro.

It‚Äôs a good idea to keep your branches up to date with their intended target branch. This reduces the chance of a conflict and makes sure your work in progress now has changes your team have released (ex: an important bug fix).

Get your the-intro branch ‚Äúup to date‚Äù with the latest changes from main by merging main into the-intro:

git switch the-intro
git merge main

That‚Äôs it! Your branch is now fully ‚Äúup to date‚Äù.

üîé In the real world you might use git pull or git pull --rebase to get up to date. pull is a git command that is used to merge from a remote branch ‚Äì that is, a branch that exists on your team‚Äôs remote (GitHub, GitLab, Gerrit, etc) ‚Äì for now, we aren‚Äôt using any remote branches.

We‚Äôre all done, so now you can merge the-intro into main to release it (and then delete it):

git switch main

git merge the-intro

git branch -d the-intro

Did you notice what type of merge it performed?

You should have seen a fast forward merge. Any idea why?

It‚Äôs nice but not required to ‚Äúget up to date‚Äù (merge main into your feature branch) before you release (merge into main). We‚Äôre practicing getting up to date - something you‚Äôll likely do many times a day on your freature branches while you work on them.

Bonus #2

Some teams prefer to require a ‚Äúmerge commit‚Äù when any branch is released, that is, merged into the mainline. Merge commits provide an easy way to revert a merge, rolling back a release.

Undo the fast-forward merge that just happened and then re-do the merge, but this time require a ‚Äúmerge commit‚Äù

Need a hint? To undo a fast-forward merge, you‚Äôre going to use reset. You need to determine the right commit id to ‚Äúreset to‚Äù. It‚Äôs a fast-forward merge, so just using reset HEAD^ might not be sufficient.

The solution... 
git reset ORIG_HEAD --hard


‚ö†Ô∏è A word of warning: reset should only really be used on branches and commits that are not yet used by your team.

üîé This is because reset modifies the graph (typically by going backwards or traversing history) versus commit and merge, which are always adding to the graph and moving history forward. git is not able safely synchronize (merge in) changes due to a reset and so they are generally avoided on shared branches like main or master.

revert is the more appropriate command to undo a change on a shared branch.
